# client-demo-factory

Фабрика персонализированных демо для российских B2B-компаний.

Для каждой компании из проверенной базы создается маленькая автономная
демо-страница — «вот каким мог бы быть ваш сценарий». Страница публикуется
по индивидуальной ссылке через GitHub Pages, к ней готовится письмо и
сообщение в мессенджер. **Отправляет всегда человек, вручную.**

---

## Быстрый запуск

```bash
cd client-demo-factory
claude
```

Затем первый запрос:

```
Прочитай CLAUDE.md и обработай первую партию клиентов по подготовленному основному промпту.
```

## 1. Что делает проект

| Этап | Что происходит | Кто делает |
|---|---|---|
| 1 | Читает факты о компании из проверенной базы | Claude |
| 2 | Создает автономное демо `index.html` | Claude |
| 3 | Копирует демо в `docs/<slug>/` | Claude |
| 4 | Проверяет HTML, секреты, мобильную версию | Claude |
| 5 | Публикует через GitHub Pages, проверяет HTTP 200 | Claude |
| 6 | Готовит письмо и сообщение | Claude |
| 7 | **Проверяет ссылку глазами** | **Человек** |
| 8 | **Отправляет клиенту** | **Человек** |

Claude никогда не выполняет шаги 7 и 8.

## 4. Где хранятся письма

Два места, и это осознанно:

- `clients/<client-id>-<slug>/messages/` — черновики, созданные при обработке;
- `outreach/` — итоговые тексты, готовые к отправке.

Файл с суффиксом `_ready.txt` — это версия, где `{{DEMO_URL}}` уже заменен
на реальную ссылку. Исходная версия сохраняется рядом, не перезаписывается.

Шаблоны — в `templates/outreach/`.

## 6. Как обработать первую партию

```
Прочитай CLAUDE.md и обработай первую партию клиентов по подготовленному основному промпту.
```

По умолчанию партия — 3 клиента (`defaults.first_batch_size` в конфиге).
Начинать имеет смысл с приоритетов 1–3: у них самый высокий lead score.

Не запускай сразу все 10. Проверь качество на трех.

## 7. Как проверить страницу локально

Просто открыть файл в браузере:

```bash
start docs/<slug>/index.html          # Windows
```

Демо автономное — оно обязано работать с диска, без сервера и без сети.
Если не работает — это баг демо, а не окружения.

Если хочется проверить именно так, как увидит клиент (по HTTP):

```bash
npx serve docs
# затем открыть http://localhost:3000/<slug>/
```

Проверить структуру и безопасность:

```bash
bash scripts/validate-project.sh
bash scripts/check-published-html.sh docs/
```

## 8. Как опубликовать изменения

```bash
git status                                   # что реально уйдет
bash scripts/validate-project.sh
bash scripts/check-secrets.sh
bash scripts/check-published-html.sh docs/

git add docs/ outreach/ reports/
git commit -m "Publish personalized demos"
git push origin main
```

Порядок важен: сначала проверки, потом commit, потом push.
Если любая проверка вернула ошибку — push не делается.

## 9. Как проверить GitHub Pages

После push деплой занимает 1–2 минуты.

```bash
gh run list --limit 5                        # статус деплоя
curl -I https://<username>.github.io/client-demos/<slug>/
```

Ожидаемый ответ — `HTTP/2 200`.

Проверять нужно **в браузере без авторизации GitHub** (приватное окно).
Страница, которая открывается у вас, может не открываться у клиента.

`404` сразу после первого push — обычно означает, что Pages еще не
включен в Settings или деплой не закончился.

## 10. Как подключить собственный домен

1. Заполнить `custom_domain` в `config/project.local.json`.
2. Создать файл `docs/CNAME` с одной строкой — доменом без протокола:
   ```
   demo.example.ru
   ```
3. У регистратора добавить DNS-запись:
   ```
   Type: CNAME
   Host: demo
   Value: <username>.github.io
   ```
4. В Settings → Pages дождаться проверки домена и включить
   **Enforce HTTPS**.
5. Проверить:
   ```bash
   nslookup demo.example.ru
   curl -I https://demo.example.ru/
   ```

DNS расходится до 24 часов. Не считать домен рабочим, пока `curl` не
вернул 200 по HTTPS.

Без своего домена файл `CNAME` создавать не нужно.

## 11. Что обязательно проверить перед отправкой

Это последний рубеж. Ошибка здесь уходит живому человеку.

- [ ] Ссылка открывается **в приватном окне**, без авторизации GitHub.
- [ ] Открывается на телефоне.
- [ ] Название компании написано правильно.
- [ ] В демо нет выдуманных фактов: цен продукции, заявок, CRM, партнеров.
- [ ] Есть подпись «Персональная демонстрационная концепция».
- [ ] Нет незамененных `{{...}}`.
- [ ] В письме `{{DEMO_URL}}` заменен на реальную ссылку.
- [ ] Ссылка ведет на **этого** клиента, а не на соседнего.
- [ ] Признаки безопасности в письме соответствуют реальности страницы.
- [ ] Страница не выглядит как официальный сайт компании.
- [ ] Адрес получателя взят из проверенного источника.

## 12. Что нельзя хранить в репозитории

- исходные базы клиентов: `.xlsx`, `.csv`, `.json` с контактами;
- `AI_sales_pipeline_Russia.pdf` и подобные внутренние документы;
- `config/project.local.json`;
- токены, API-ключи, пароли, cookies, OAuth secrets;
- приватные SSH-ключи (`*.pem`, `*.key`, `*.p12`, `*.pfx`);
- содержимое `source/processed/` и `archive/`.

Все перечисленное уже закрыто в `.gitignore`.
Проверить перед commit:

```bash
bash scripts/check-secrets.sh
git status --short
```
