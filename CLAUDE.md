# client-demo-factory — постоянная инструкция проекта

Этот файл читается в начале каждой сессии. Он имеет приоритет над догадками и
привычными шаблонами. Если инструкция противоречит удобству — следуй инструкции.

---

## 1. Назначение

Проект создает **персонализированные статические демо** для российских B2B-компаний
и публикует их через **GitHub Pages**.

Логика работы:

1. Берем проверенные факты о компании из локальной базы клиентов.
2. Создаем маленькое автономное демо — «вот каким мог бы быть ваш сценарий».
3. Публикуем демо по индивидуальной ссылке со случайным slug.
4. Готовим письмо и сообщение в мессенджер.
5. Человек проверяет вручную и отправляет сам.

Цель демо — получить ответ и назначить разговор. Не продать в письме.

---

## 2. Основные источники

Читай в этом порядке:

| Источник | Путь | Роль |
|---|---|---|
| Локальная конфигурация | `config/project.local.json` | имена, ссылки, контакты отправителя |
| Проверенная таблица клиентов | `source/incoming/Claude_clients_Russia_verified.xlsx` | **основной источник фактов** |
| JSON клиентов | `source/incoming/clients_verified.json` | те же данные, машиночитаемо (10 записей) |
| Client brief | `clients/<client-id>-<slug>/analysis.md` | разбор конкретного клиента |
| Предыдущие QA-отчеты | `reports/DEMOS_QA_REPORT.md` | что уже проверено и что сломано |
| Основной промпт | `source/incoming/CLAUDE_START_PROMPT.md` | сценарий пакетной обработки |

Если таблица и JSON расходятся — **таблица важнее**, JSON считается производным.

Если у компании есть собственный сайт — он приоритетный источник фактов.

## 3. Правила достоверности

Это самое важное правило проекта. Демо показывается реальной компании,
которая знает про себя правду. Одна выдуманная деталь убивает доверие
и сделку.

### Запрещено придумывать

- имена;
- контакты;
- должности;
- партнеров;
- клиентов компании;
- количество заявок;
- CRM;
- финансовые показатели;
- средний чек;
- внутренние процессы;
- цены продукции;
- сроки производства.

### Неизвестное должно оставаться неизвестным

Не заполняй пробел правдоподобной догадкой. Варианты поведения:

- убрать блок;
- заменить нейтральной формулировкой;
- пометить `⚠️ требует согласования`;
- спросить пользователя.

### Условные числа

Любые расчетные цифры внутри демо помечай:

```
демонстрационные данные
```

или

```
предварительный расчет, не оферта
```

Цены **продукции клиента** не выдумываются никогда, даже с пометкой.
Цены **наших услуг** берутся только из полей `implementation_price_rub`
и `monthly_support_rub` в базе клиентов.

## 4. Технические требования демо

- чистый HTML, CSS и JavaScript;
- без API;
- без backend;
- без авторизации;
- без платежей;
- без скачивания;
- без внешней отправки данных;
- мобильная адаптация;
- `noindex`;
- демонстрационная маркировка;
- один законченный сценарий;
- блок «В полной версии».

### Что это значит на практике

Демо — **один автономный файл** `index.html` со встроенными стилями и скриптами.
Он должен открываться двойным кликом с диска и работать без сети.

Запрещено:

- `<script src="https://...">` — внешние библиотеки;
- `fetch`, `XMLHttpRequest`, `navigator.sendBeacon`;
- `<form action="https://...">` — реальная отправка;
- `<input type="password">`;
- ссылки на `file://`, `C:\`, `/Users/`;
- шрифты и картинки с CDN.

Обязательный мета-набор:

```html
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1">
<meta name="robots" content="noindex, nofollow, noarchive">
```

Обязательная подпись:

```
Персональная демонстрационная концепция. Не является официальным сайтом компании.
```

### Один законченный сценарий

Демо делает **одно** действие до конца. Все остальное — в блок «В полной версии».
Лучше один работающий сценарий, чем пять кнопок-заглушек.

## 5. Безопасность

- не публиковать исходные базы клиентов;
- не публиковать токены;
- не публиковать закрытые данные;
- не отправлять сообщения автоматически;
- всегда требовать ручную проверку перед отправкой;
- не утверждать успешную публикацию без HTTP-проверки.

### Что никогда не попадает в Git

- `source/incoming/*.xlsx | *.json | *.pdf | *.csv | *.zip`
- `config/project.local.json`
- `source/processed/`, `archive/`
- любые токены, ключи, пароли, cookies

Перед каждым commit запускай `scripts/check-secrets.sh`.

### Публичная главная страница

`docs/index.html` **не должна** содержать:

- список клиентов;
- названия компаний;
- ссылки на демо;
- контактную базу;
- внутренние отчеты.

Демо находят только по прямой индивидуальной ссылке.

### Про slug

Slug должен быть неугадываемым: `<company-slug>-<random-suffix>`, например
`mdo-a7k29q`. Это не защита, а гигиена — чтобы демо одного клиента не
находилось перебором из демо другого.

## 6. Git-процесс

Порядок строгий:

1. проверить diff;
2. запустить валидацию;
3. проверить секреты;
4. коммитить только безопасные файлы;
5. push только после успешных проверок;
6. после публикации проверять HTTP 200;
7. сохранять отчет.

```bash
git status
git diff
bash scripts/validate-project.sh
bash scripts/check-secrets.sh
bash scripts/check-published-html.sh docs/
git add <только проверенные пути>
git commit -m "..."
git push origin main
```

Никогда не делай:

- `git add -A` без проверки `git status`;
- `git push --force`;
- перезапись удаленной истории;
- push при найденных секретах.

## 7. Письма

### Не писать

```
Это не фишинг.
```

Фраза «это не фишинг» вызывает ровно то подозрение, которое пытается снять.
Так пишут только фишеры.

### Использовать проверяемые признаки

Вместо оправданий — факты, которые получатель может проверить сам за 5 секунд:

- обычная HTTPS-страница;
- без авторизации;
- без паролей;
- без оплаты;
- без скачивания файлов;
- без передачи введенных данных, **только если это подтверждено**.

Последний пункт — утверждение о факте. Пиши его **только** если ты
проверил HTML и там действительно нет отправки данных. Если не проверил —
не пиши.

### Тон

- один вопрос в конце, не три;
- без «уникальное предложение», «революционный», «взрывной рост»;
- персональное наблюдение должно быть про **их** сайт, а не про нас;
- ссылка — рано, в первых строках;
- подпись — реальный человек из `config/project.local.json`.

## 8. Статусы

Единый список. Не изобретать новые.

```
Новый
Готов к обработке
Демо создается
Требуется проверка
Опубликовано — требуется ручная проверка
Готово к отправке
Сообщение отправлено
Получен ответ
Назначен созвон
Пилот предложен
Пилот запущен
Предложение отправлено
Продажа
Отказ
Отложено
```

Ключевые переходы:

- `Опубликовано — требуется ручная проверка` → `Готово к отправке`
  **только после того, как человек открыл ссылку глазами**. Claude не
  переводит в этот статус сам.
- `Сообщение отправлено` ставит **только человек**, который нажал «отправить».
  Claude никогда не отправляет сообщения.

---

## 9. Что Claude не делает без отдельной команды

- не создает новые демо;
- не переделывает готовые демо;
- не меняет концепцию, тексты, механику, дизайн, цены и данные клиента;
- не отправляет письма и сообщения;
- не публикует страницы с клиентами до завершения настройки;
- не коммитит исходные базы;
- не устанавливает тяжелые зависимости;
- не меняет глобальную конфигурацию Git;
- не утверждает, что Pages/DNS/HTTPS настроены, без проверки.

---

## 10. Быстрая проверка перед push

```bash
bash scripts/validate-project.sh      # структура и конфиги
bash scripts/check-secrets.sh         # секреты
bash scripts/check-published-html.sh docs/   # безопасность страниц
git status                            # что реально уйдет
```

Все три должны вернуть `0`. Иначе push не делается.
